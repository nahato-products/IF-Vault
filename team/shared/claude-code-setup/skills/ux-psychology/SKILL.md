# UX for App Development

## When to Apply

- UI設計・実装・レビュー全般（コンポーネント、フロー、バリデーション、レスポンシブ）
- AI機能のUX設計（チャットUI、レコメンド、ストリーミング応答）
- ニューロダイバーシティ・アクセシビリティの認知面での対応

## When NOT to Apply

バックエンドのみのロジック、DB設計、CI/CD、テストコード（テストUI自体でない限り）、API設計、描画以外のパフォーマンス最適化

## Decision Tree

```
情報量多い？→Yes: 選択肢多い→#2 Hick+#3 段階的開示 / 少ない→#1 認知負荷+#25 パレート
情報量少ない→FB不足？→応答400ms超→#9 ドハティ / 以内→H1 可視性
FB十分→入力問題？→誤操作リスク→#14 意図的摩擦 / 低リスク→#13 ポステル
入力問題なし→AI機能？→Yes→#28 AI透明性+#29 AI制御 / No→#16 ゲシュタルト+#15 視覚的階層
```

## Part 1-3: 認知・フィードバック・入力 [CRITICAL]

| # | 原則 | キールール |
|---|------|----------|
| 1 | 認知負荷 | 1画面の選択肢5個以下。関連情報をチャンクに。進捗インジケーター |
| 2 | Hick's Law | 20項目超→オートコンプリート。設定は基本/詳細分離。プラン選択3つ以内 |
| 3 | 段階的開示 | 高度な設定は折りたたみ。必須項目のみ初期表示。ウィザード形式 |
| 4 | バナーブラインドネス | 重要通知を広告風にしない。インラインメッセージ優先 |
| 5 | 選択的注意 | F型/Z型の視線流に沿って配置。空白で重要要素を浮かせる |
| 6 | 系列位置効果 | ナビ/タブバーの最初と最後に最重要項目 |
| 7 | フォン・レストルフ | CTAは異色・異サイズで孤立。孤立要素は1画面1-2個まで |
| 8 | Jakob's Law | 業界標準パターンに従う。大幅リニューアルは段階的に |
| 9 | ドハティ閾値 | 応答400ms超→即座にスケルトン/スピナー。楽観的UI更新 |
| 10 | レイバーイリュージョン | 処理過程を見せると価値を感じる。AI活用→#28 |
| 11 | リアクタンス | 強制登録・ポップアップ乱発は反発。スキップ/後でを必ず用意 |
| 12 | デフォルトバイアス | デフォルト=最も安全で推奨。ダークパターン禁止 |
| 13 | ポステルの法則 | 入力は寛容に受付（全角半角・ハイフン混在OK）、出力は統一 |
| 14 | 意図的摩擦 | 不可逆操作のみ確認ダイアログ。通常操作に摩擦はUX劣化 |

## Part 4: フローと導線 [HIGH]

| # | 原則 | キールール |
|---|------|----------|
| 15 | 視覚的階層 | 見出し比1.25倍以上。色3色以内。余白=「多い→重要」「近い→関連」 |
| 16 | ゲシュタルト | 近接: 関連密着、グループ間離す。類似: 同機能同スタイル。閉合: 端を少し見せる。図と地: モーダル背景暗く |
| 17 | ビジュアルアンカー | 固定ヘッダー/フッター。パンくずリスト。スクロール連動ナビ |
| 18 | リアクティブオンボーディング | 初回は最低限ツアー。初使用時ヒント。Empty Stateで次アクション案内 |
| 19 | ゴール勾配効果 | プログレスバーで現在地表示。「あと1ステップ」 |
| 20 | ツァイガルニク効果 | 「70%完了」表示。中断→「続きから再開」。長いフォームは一時保存 |
| 21 | Peak-End Rule | 完了画面にお祝い演出。エラーをピークにしない。復帰を簡単に |

## Part 7: AI UX [HIGH]

### 28. AI透明性と信頼

- 「AIが生成しました」ラベル必須。信頼度を示す
- 判断根拠を説明可能に。限界を正直に伝える
- 信頼の段階的構築: 精度を見せてから権限を広げる

### 29. AI制御とエラー回復

- 「提案→確認→適用」フロー。自動適用しない
- 「AIオフ」「再生成」「元に戻す」を常に提供
- フォールバック: AI失敗→手動操作へシームレス切替

## Part 8: ニューロダイバーシティ [CRITICAL]

WCAG技術実装は web-design-guidelines 参照。ここでは認知多様性の補足:

- **ADHD**: 通知最小限。フォーカスモード。タスク小分割 → #1
- **ディスレクシア**: 左揃え統一。行間1.5+。1行60文字以下
- **自閉症**: パターン一貫性徹底。急な変化を避ける
- `prefers-reduced-motion` + `prefers-contrast` + `prefers-reduced-transparency` 全サポート

## Part 9: Experience Differentiation [HIGH]

同じコンポーネントライブラリでも**体験設計**でアプリの個性は出せる。見た目は `design-token-system` の5軸で。

| 要素 | AI slop | 差別化 |
|------|---------|--------|
| オンボーディング | 3枚スライドツアー | ドメイン固有の初回体験 |
| Empty State | 「データがありません」 | プロダクト価値を伝える導線(#18) |
| エラー回復 | 「エラーが発生しました」 | ドメイン固有の修正提案(H9+#13) |
| 成功の瞬間 | チェックマーク | ドメインに合った祝福(#21) |
| 待ち時間 | スピナー | 世界観に合った演出(#10) |
| マイクロコピー | 「送信する」 | ブランドの声(`natural-japanese-writing`) |

**設計前チェック**: (1)ドメイン固有の瞬間は何か (2)ユーザーの感情状態は (3)競合と同じ体験になってないか

## UX Review Checklist

- [ ] 1画面の選択肢5個以下(#1) / 400ms超でスケルトン(#9)
- [ ] 不可逆操作に確認ステップ(#14) / 入力は寛容に受付(#13)
- [ ] CTA孤立+十分なサイズ(#7+#22) / エラーに修正方法(H9)
- [ ] ショートカット提供(H7) / AI生成ラベル+再生成(#28-#29)
- [ ] prefers-reduced-motion/contrast/transparency対応 / Jakob's Law準拠(#8)
