# Brainstorming — Reference

ブレインストーミングスキルで使うアイデア発想テンプレート、意思決定フレームワーク、優先順位付け手法のリファレンス集。

---

## Process（再掲）

```
1. Context  — files, docs, commits
2. Clarify  — purpose, constraints, success criteria
3. Explore  — 2-3 approaches with trade-offs
4. Present  — approval per section
5. Document — docs/plans/YYYY-MM-DD-<topic>-design.md
6. Plan     — EnterPlanMode or proceed
```

## Ideation Techniques

### SCAMPER

既存のアイデアや機能を7つの視点で変形させる手法。機能改善・リデザインに最適。

| 視点 | 問い | 例（チャット機能の場合） |
|------|------|--------------------------|
| **S**ubstitute（代替） | 何を別のもので置き換えられる？ | WebSocket → SSE に置き換え |
| **C**ombine（結合） | 2つの機能を合体できる？ | チャット + タスク管理を統合 |
| **A**dapt（適応） | 他の分野のアイデアを借りられる？ | Slack のスレッド構造を採用 |
| **M**odify（変更） | 形・サイズ・頻度を変えたら？ | メッセージにリアクション追加 |
| **P**ut to other use（転用） | 別の目的に使えない？ | チャットログを FAQ に転用 |
| **E**liminate（削除） | 何を取り除ける？ | 既読表示を削除してシンプルに |
| **R**everse（逆転） | 順序や役割を逆にしたら？ | ユーザーが質問 → AIが質問する形に |

**使い方**: 改善対象の機能を1つ選び、7つの問いを順に当てはめる。全部に答える必要はなく、刺さったものだけ深掘りする。

### How Might We（HMW）

課題を「どうすれば〜できるだろう？」の形に変換して、解決策の発散を促すテンプレート。

```
テンプレート:
  「どうすれば [ユーザー] が [状況] で [目的] を達成できるだろう？」

例:
  - どうすれば初回ユーザーが迷わずに最初のプロジェクトを作成できるだろう？
  - どうすれば管理者がチームの進捗を一目で把握できるだろう？
  - どうすればモバイルユーザーが片手操作でタスクを完了できるだろう？
```

**コツ**:
- 広すぎず狭すぎない粒度にする（「UXを改善するには？」は広すぎ、「ボタンの色は？」は狭すぎ）
- 1つの課題から HMW を3〜5個出して、最も可能性が広がるものを選ぶ
- 制約を含めるとより実用的（「既存のDBスキーマを変えずに〜」等）

### Crazy 8s（高速スケッチ）

8分間で8つのアイデアを出す。質より量で発散させるテクニック。

```
進め方:
  1. テーマを1つ決める（例:「ダッシュボード画面のレイアウト」）
  2. 1アイデア = 1分。計8分で8案出す
  3. 各アイデアは 2-3行の箇条書き or ASCII図で十分
  4. 評価は後。出す段階では批判禁止
  5. 8案から上位2-3案を選んで深掘り

Claude Code での適用:
  - 「Crazy 8s 形式で8案出して」と依頼
  - 各案に [コスト: 低/中/高] [インパクト: 低/中/高] をラベル付け
  - 上位案を選んでから設計フェーズに入る
```

## Decision Matrix（意思決定マトリクス）

複数のアプローチを定量的に比較する。brainstorming の Step 3「Propose 2-3 approaches」で使う。

### テンプレート

```markdown
## 比較: [テーマ]

| 評価軸 (重み) | Option A: [名前] | Option B: [名前] | Option C: [名前] |
|---------------|-------------------|-------------------|-------------------|
| 実装コスト (3) | ◎ 低い (9) | ○ 中程度 (6) | △ 高い (3) |
| パフォーマンス (2) | ○ 十分 (4) | ◎ 最速 (6) | ○ 十分 (4) |
| 保守性 (3) | ◎ シンプル (9) | △ 複雑 (3) | ○ 普通 (6) |
| 拡張性 (2) | △ 限定的 (2) | ◎ 高い (6) | ○ 普通 (4) |
| **合計** | **24** | **21** | **17** |

→ [推奨] Option A: 実装コストと保守性で優位。拡張性は将来必要になった時点で対応可能。
```

### スコアリングルール

| 記号 | スコア | 意味 |
|------|--------|------|
| ◎ | 3 | 優れている |
| ○ | 2 | 十分 |
| △ | 1 | 懸念あり |

最終スコア = 各軸の（スコア × 重み）の合計

**重みの目安**:
- 3: プロジェクトの成否に直結する軸
- 2: 重要だが妥協可能な軸
- 1: あれば嬉しい程度の軸

## Feature Prioritization

### MoSCoW 法

要件を4カテゴリに分類。MVP定義やスコープ決めに最適。

| カテゴリ | 意味 | 判断基準 |
|----------|------|----------|
| **Must** | 必須 | これがないとリリースできない / ユーザーの根本的な課題を解決しない |
| **Should** | 重要 | 強く望まれるが、なくてもコア機能は動く |
| **Could** | あると嬉しい | リソースに余裕があれば対応 |
| **Won't** (this time) | 今回はやらない | 将来対応の可能性はあるが、今は明示的にスコープ外 |

```markdown
## MoSCoW: [機能名]

### Must（必須）
- [ ] ユーザー認証（メール + パスワード）
- [ ] プロジェクト CRUD
- [ ] 基本的なエラーハンドリング

### Should（重要）
- [ ] OAuth ログイン（Google）
- [ ] メール通知
- [ ] ダッシュボードの集計表示

### Could（あると嬉しい）
- [ ] ダークモード
- [ ] CSV エクスポート
- [ ] キーボードショートカット

### Won't（今回はやらない）
- [ ] モバイルアプリ
- [ ] 多言語対応
- [ ] リアルタイムコラボ
```

### RICE スコアリング

数値で優先順位を決める。バックログの整理やロードマップ策定に向いている。

```
RICE スコア = (Reach × Impact × Confidence) / Effort
```

| 要素 | 説明 | スケール |
|------|------|----------|
| **R**each | 一定期間に影響するユーザー数 | 実数（例: 月間500人） |
| **I**mpact | 1ユーザーあたりの影響度 | 3=大, 2=高, 1=中, 0.5=低, 0.25=最小 |
| **C**onfidence | 見積もりの確信度 | 100%=高, 80%=中, 50%=低 |
| **E**ffort | 必要な人月 | 実数（例: 2人月） |

```markdown
## RICE: バックログ優先順位

| 機能 | Reach | Impact | Confidence | Effort | Score | 順位 |
|------|-------|--------|------------|--------|-------|------|
| 検索フィルタ改善 | 800 | 2 | 80% | 1 | 1280 | 1 |
| ダッシュボード刷新 | 500 | 3 | 50% | 3 | 250 | 3 |
| CSV エクスポート | 200 | 2 | 100% | 0.5 | 800 | 2 |
| AI レコメンド | 300 | 3 | 50% | 5 | 90 | 4 |
```

## Example: Brainstorming Session Output

以下は実際のブレストセッションの出力イメージ。

```markdown
# Design: タスク管理アプリのリマインダー機能
日付: 2026-02-24

## 背景
既存のタスク管理アプリにリマインダー機能を追加したい。
ユーザーからの要望が最も多い機能（月間リクエスト: 47件）。

## 明確化した要件
- 対象: 全タスクに対して任意でリマインダーを設定可能
- 通知手段: ブラウザ Push 通知（Phase 1）、メール（Phase 2）
- 繰り返し: なし（Phase 1）、日次/週次（Phase 2）
- タイムゾーン: ユーザー設定に従う

## 検討したアプローチ

### [推奨] Option A: Supabase Edge Functions + pg_cron
- コスト: 低（既存インフラ内で完結）
- 保守性: 高（Supabase に寄せて一元管理）
- 制約: 1分単位の精度（pg_cron の制限）

### [代替] Option B: Vercel Cron + Upstash Queue
- コスト: 中（Upstash の追加料金）
- 保守性: 中（外部サービスが増える）
- 利点: 秒単位の精度、リトライ機構が堅牢

### 不採用: Option C: クライアントサイド setTimeout
- 理由: タブを閉じると動作しない。UX として信頼性が低い

## MoSCoW
- Must: 単発リマインダー設定/削除、ブラウザ Push 通知
- Should: リマインダー一覧画面、通知履歴
- Could: 繰り返し設定、メール通知
- Won't: Slack/LINE 連携（Phase 3 以降）

## 決定
→ Option A で進める。pg_cron の1分精度はリマインダー用途では十分。
→ 実装計画は EnterPlanMode で作成。
```

## Principles（再掲）

- **Design before implementation**: どんなに小さくても設計→承認→実装の順
- **One question at a time**: 質問は1つずつ。複数選択肢を添えて
- **YAGNI ruthlessly**: 「将来使うかも」は今は切る
- **2-3 approaches always**: 1案だけで決めない。比較して選ぶ
- **Incremental validation**: セクションごとに承認を取る
